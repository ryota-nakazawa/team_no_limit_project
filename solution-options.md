# フェーズ2: 解決手段の選定

## 比較候補

| 候補 | 概要 | 利点 | 弱み |
| --- | --- | --- | --- |
| GPTs 単体 | 用途別 GPT を作成して利用 | 構築が速い | 業務カテゴリの一元ナビゲーションが弱い |
| エージェント型 | 状況把握からタスク分解まで自律支援 | 将来拡張性が高い | PoC には過剰で設計コストが高い |
| 軽量 Web アプリ | カテゴリ UI とプロンプトをまとめて提供 | 使い分けが明確で PoC 向き | 外部連携は別途実装が必要 |
| ワークフロー自動化 | 定型業務をトリガー起点で自動処理 | 省力化効果が高い | 前提データや運用設計が必要 |

## 最小実現手段
- 第一候補は「軽量 Web アプリ + カテゴリ別プロンプトテンプレート + チャット UI」。

## 選定理由
- 課長にとって「何に使えるか」が一目で分かることが最優先であり、カテゴリナビゲーションを持つ UI が必要。
- 既存 React アプリを活用できるため、PoC の立ち上がりが最も速い。
- 将来的に GPTs やワークフローへ展開する際も、用途整理の母艦として流用できる。

## トレードオフ
- 自律実行や外部システム連携は限定的になる。
- 現段階では、利用者が必要情報を入力する前提が残る。
- 定型業務の完全自動化よりも、判断支援と下書き支援に寄る。

## 現時点の結論
- PoC は軽量アプリで成立確認を行い、その後の検証結果で GPTs 分割やワークフロー連携を判断する。
